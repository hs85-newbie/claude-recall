## 코딩 핵심 원칙

- **소프트웨어의 적은 버그가 아니라 복잡성이다.** 모든 설계·구현 결정에서 복잡성을 최소화하는 선택을 우선한다. 버그는 복잡성의 증상이며, 복잡성을 줄이면 버그의 발생 여지 자체가 줄어든다. 기능 추가·추상화·의존성 도입 시 "이게 전체 복잡성을 늘리는가?"를 먼저 묻고, 늘린다면 그만한 가치가 있는지 명시적으로 정당화한다.
- 투기적 추상화 금지 — 지금 필요한 것만 구현
- 검증은 시스템 경계(사용자 입력, 외부 API)에서만
- pnpm 기본, TypeScript 우선, any 타입 금지
- DRY: 동일 로직 2회 이상 등장 시 즉시 함수 추출
- 함수 최대 30줄 / 파일 최대 300줄 (초과 시 분리)
- 구현 전 검증 기준(테스트 또는 성공 조건) 먼저 정의
- 새 작업 시작 전 무관한 이전 컨텍스트는 /clear로 리셋

## 테스트 원칙

- 새 기능: 단위 테스트 필수 / 외부 연동: 통합 테스트 필수
- 커버리지 80% 미만 PR 머지 금지
- 테스트 없는 코드 = 완성되지 않은 코드

## 에러 처리 원칙

- 외부 API 호출: try/catch + 재시도 최대 3회 (지수 백오프)
- 빈 catch 블록 금지 — 반드시 로그 또는 상위 전파
- 사용자 노출 메시지: 기술 정보 제거, 친화적 표현만
- 상세 규칙: @docs/error-handling.md

## 메모리 관리 규칙

MEMORY.md는 프로젝트별 카테고리로 구분하여 관리한다:

- `## 공통` — 사용자 프로필, 전역 피드백
- `## [프로젝트명]` — 해당 프로젝트 전용 메모리
- `## 파이프라인 피드백` — 워크플로우/dispatch 관련 피드백

메모리 저장 시:

- **저장 직전 자가검증 필수**: "이게 다음 세션에서도 *자동으로* 적용돼야 하는 영구 룰/사실인가?" → YES면 메모리, NO(진행 상태·잔여 작업·세션 결정)면 gstack `/checkpoint`로 위임
- 해당 프로젝트 카테고리에 배치
- 다른 프로젝트에도 적용되는 피드백은 `공통` 또는 `파이프라인 피드백`에 배치
- 세션 시작 시 현재 작업 프로젝트의 카테고리만 우선 참조
- **본문 표적 Read 룰** (2026-06-09 추가): 세션 시작 시엔 MEMORY.md 인덱스(한 줄 요약)만 로드한다. 단 **되돌리기 어려운 주제**(전략·아키텍처·발행정책·비용·범위 결정)에 진입해 사용자에게 질문·제안하기 **직전**, 그 주제에 매칭되는 메모리 **본문을 Read한 뒤** 진행한다.
  - **표적/게으른 Read**: 매칭되는 본문 1~3개만 읽는다. 전체 본문 일괄 로드 금지(토큰 낭비). 단순 조회는 인덱스로 충분 → 본문 Read 불필요.
  - 근거: 인덱스만 보고 이미 확정된 결정을 재논의하면 여러 턴 + 잘못된 가정 위험이 표적 본문 Read(~1k 토큰)보다 훨씬 비싸다.
