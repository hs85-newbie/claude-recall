# /verify-layer — 결과물 검증 (비판 패널 + 3층 루브릭)

AI가 생산을 대신할수록 사람의 메인 작업은 **검증**이다. 전부 검증하면 지친다 → **중간상태는 넘기고 결과물만 철저히**, 가능한 건 자동화한다. 핵심 통찰: **검증은 별개 인스턴스로** 한다(같은 세션에 "검증도 해"하면 자기 말을 옹호해 일부만 잡는다).

## 절차

### 1. 대상·성공조건 확정
- 무엇을 검증하나(코드·번역·생성콘텐츠·설계·납품물).
- **성공 조건이 정의돼 있나?** 없으면 그것부터 한 줄로 정의(검증 기준 없는 검증은 무의미).
- 이 결과물이 **외부로 나가나?**(발행·머지·납품·push) → 그렇다면 마지막에 컨펌 게이트 필수.

### 2. 3층 루브릭 적용 (가능한 층은 자동화)
- **Binary (통과/실패)** — 객관 체크를 Bash로 자동: 테스트·린트·빌드·타입체크·스키마검증·존재여부. 실패는 즉시 결함.
- **Quantitative (수치)** — 측정 가능 지표: 처리량·latency·비용·변경률·커버리지. **번역·생성물이면 사람이 만든 정답셋(human-gold) 대비 정량 비교를 기본 방법으로**(LLM 자가심판보다 우선).
- **Qualitative (주관 품질)** — 자동화 안 되는 부분만 비판 패널(아래)로.

### 3. 비판 패널 (별개 인스턴스 N개 병렬)
Qualitative 층에서, `critic` 에이전트를 **서로 다른 렌즈로 동시에** 소환한다(Agent 도구, 한 메시지에 복수 호출 → 병렬). 대상 성격에 맞는 렌즈를 고른다:
- 코드 → 정확성/논리 · 보안 · 복잡성 · 재현성
- 번역/콘텐츠 → 내용 충실성 · 자연스러움 · 사용자 노출 · 비용
- 설계 → 복잡성 · 재현성 · 비용

각 critic은 통과시키려 하지 않고 **반증**한다. (LLM-as-judge 1~5 루브릭이 필요하면 한 critic에게 점수화를 맡긴다.)

### 4. 수렴 루프 (최대 2회)
1. 패널 발견을 모아 **중복 제거**.
2. **타당한 발견만 메인 세션에서 재검증**한다 — critic이 맞는지 직접 확인(거짓 양성 거르기). 추정 말고 1차 소스/실행으로.
3. 확정된 결함이 있으면 보고(고칠지는 사용자/메인 판단). 새 라운드가 필요하면 1회 더 패널을 돌린다.
4. **2회로 끝낸다** — 무한 루프 금지. 2회 후에도 남은 불확실은 "미해결"로 표기.

### 5. 결과 보고
```
## 검증 결과: <대상>
성공조건: <무엇을 기준으로 봤나>

Binary:   통과 N / 실패 M  (실패 항목)
Quant:    <지표=값> (사람-골드 대비 결과가 있으면 여기)
Qual:     critic 패널 N개 → 확정 결함 K개

확정 결함(재검증 통과분):
- [심각도] <문제> — 근거 — 제안

미해결/미확인:
- 

외부 발행 여부: <예/아니오> → 예면: ⚠️ 컨펌 게이트 — 사람 승인 전까지 발행 금지
```

## 철칙
- **빈 검증 금지.** "통과"라면 *무엇을 어떻게 확인했는지* 밝힌다. 확인 안 한 층은 "미적용"이라 명시.
- **거짓 양성 차단.** critic 발견을 그대로 결함으로 올리지 말고 메인에서 재검증한 것만 확정.
- **검증이 결정을 대신하지 않는다.** 결함·근거를 제시하되, 고치기·발행은 사용자/메인 세션의 몫. 외부로 나가는 건 항상 컨펌 게이트.
- 비용 의식: 패널 규모는 대상 위험도에 맞춘다. 사소한 산출물에 critic 6개는 과하다(2~3개). 위험·납품물일수록 늘린다.
